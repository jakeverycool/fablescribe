# Fablescribe — Pre-Planning Document

> AI-powered DM storytelling toolkit: live Discord voice transcription, DM-curated campaign memory, AI-voiced NPC responses, and a campaign-memory chatbot for recalling past events.

**Status:** Pre-planning / feasibility. Intended as the foundation for a Claude Code build document.

---

## 1. Product Summary

Fablescribe is a storytelling and memory tool for tabletop RPG Dungeon Masters. The emphasis is on **narrative and world continuity**, not stat tracking. The core value props:

1. **Live transcription** of Discord voice sessions so nothing said at the table is lost.
2. **DM-curated campaign memory** — the DM promotes important moments from the raw transcript into a canonical, searchable campaign record.
3. **AI-voiced NPC responses** generated by Claude + ElevenLabs, informed by character sheets, linked lore, and the character's history with the party.
4. **Campaign-memory chatbot** — the DM can ask natural-language questions ("what happened last time the party met Gareth?") and get answers drawn from the curated campaign memory.
5. **Flexible campaign glossary** — characters, places, factions, events, items, and lore, all stored as freeform reference entries that feed both the response generator and the chatbot.
6. **Campaign file library** — DMs can upload reference material (handouts, maps, PDFs, images, notes) and attach it to a campaign for storage. Future versions will let the AI reference uploaded text documents.

The explicit non-goal: Fablescribe does not track stats, HP, initiative, or mechanical game state. It's a storytelling and memory layer, not a virtual tabletop.

## 2. Core User Flows

### 2.1 DM starts a session
1. DM opens campaign in dashboard, clicks **Start Session**.
2. Discord bot joins the designated voice channel.
3. STT worker begins transcribing per-speaker audio into the **raw log**.
4. Live transcript streams to the dashboard via Supabase Realtime.

### 2.2 DM curates raw transcript → campaign memory
The raw log is ephemeral working material. The DM's job during/after a session is to curate the moments that matter into the **campaign memory** (the canonical record).

From a transcript selection, the DM can:
- **Add to campaign memory** — promotes selected lines (optionally with DM annotation and glossary links) as a note entry in campaign memory.
- **Generate NPC response** — opens the response flow (see 2.3).

The DM can also:
- **Create a standalone campaign memory entry** — no source transcript needed. Use case: "The party unknowingly triggered the curse by touching the altar." Or: "Combat encounter with bandits on the road — party won, took a prisoner named Kellen." This is how the DM logs things (like fights) where the full raw transcript is too noisy to ingest but the outcome matters.
- **Mark entries as public or dm_only** — dm_only entries are invisible to players, for future reveals.

### 2.3 DM generates an NPC response
1. DM selects transcript context, clicks **Generate Response**.
2. Popup: choose character, optional "additional context" freeform field.
3. Backend assembles the prompt (see §5) and calls Claude.
4. DM previews the generated text and **can edit it freely** before proceeding.
5. DM clicks **Generate Audio** → ElevenLabs streams audio using the character's assigned voice.
6. Audio is added to the **playback queue** (not auto-played).
7. DM clicks **Play** on a queue item → bot plays it in the Discord channel.
8. The full response entry (context + character + final text + audio + timestamps) is stored permanently in campaign memory and vectorized into Qdrant.

### 2.4 DM uses the campaign-memory chatbot
1. DM opens the chatbot panel in the dashboard.
2. Types a natural-language question: "What happened with Gareth the innkeeper last time?" or "Who knows about the cursed altar?" or "Remind me what the Crimson Hand wants."
3. Backend queries Qdrant against the campaign memory + glossary, retrieves relevant entries, passes them to Claude with a system prompt ("You are a helpful campaign historian. Answer based on the retrieved campaign memory.").
4. Claude returns a natural-language answer.
5. DM can follow up or drill deeper.

The chatbot only reads from **curated campaign memory and the glossary** — it never sees the raw transcript. This keeps the memory signal clean and prevents the chatbot from surfacing out-of-character table talk, mistranscribed dice rolls, etc.

### 2.5 DM controls
- **Pause STT** — temporarily halts transcription for breaks or side conversations the DM doesn't want captured at all.
- **End Session** — stops the bot, deletes raw audio files, marks session as ended. Curated memory persists.
- **Edit / soft-delete** memory entries — with re-vectorization on edit.
- **Reorder / regenerate / remove** audio queue items before playback.

### 2.6 DM manages campaign files
1. DM opens the **Files** tab in the campaign dashboard.
2. Uploads reference material: handouts, maps, homebrew PDFs, inspiration images, prep notes, etc.
3. Supported: `.docx`, `.txt`, `.md`, `.pdf`, and common image formats.
4. Each file can be renamed, tagged, and given a description.
5. Files are scoped per campaign and accessible to campaign members per role rules.
6. V1 is storage-only — no AI ingestion yet. Schema anticipates later ingestion into Qdrant.

### 2.7 Player experience
- Player signs up, enters campaign invite code, joins as a player member.
- Can view the campaign memory for sessions they attended.
- **V1 testing:** players also see the raw log. Temporary; schema supports locking this down.
- **V1:** chatbot is DM-only. Player chatbot access is a future feature.

### 3.1 Why this split

- **Node.js bot:** `@discordjs/voice` has official, maintained voice receive support. Isolates the Discord integration so a Discord API change breaks one service, not the whole app.
- **Python backend:** matches existing ExpertAI stack; good ecosystem for Claude/ElevenLabs/Deepgram SDKs, Qdrant client, and async orchestration. Runs on cheap CPU-only hosting — no GPU required.
- **Deepgram Nova-3 for STT:** cloud streaming transcription with <300ms latency, per-second billing, and Keyterm Prompting — a contextual in-context vocabulary adapter that dramatically improves accuracy on fantasy proper nouns. Eliminates the need for local GPU hardware.
- **Silero VAD on the backend:** runs on CPU, used to filter out silence before sending audio to Deepgram. This keeps Deepgram costs tied to actual speech, not session duration.
- **Qdrant:** vector search for the chatbot and historical context retrieval during response generation. Jake already runs Qdrant for ExpertAI — no new infrastructure.
- **Supabase:** auth, Postgres (source of truth), Realtime (live transcript streaming), and Storage (ephemeral audio).
- **React:** standard dashboard SPA.

### 3.2 STT pipeline

1. Bot captures per-user 48kHz stereo PCM from Discord.
2. Downsample to 16kHz mono.
3. Backend runs Silero VAD per user to detect when speech is actually happening (gates the Deepgram stream so you don't pay for silence).
4. When speech is detected for a user, open (or reuse) a Deepgram streaming WebSocket connection for that user with the campaign's keyterms attached.
5. Stream the PCM to Deepgram as it arrives.
6. Deepgram returns interim transcripts (for live display) and final transcripts (for persistence).
7. Final transcripts are written to Postgres and published via Supabase Realtime to the dashboard with per-user attribution.
8. When the session ends or a user stops speaking for a configurable window, close the Deepgram connection.

**Keyterm injection:** at session start, the backend assembles a keyterm list from the campaign's glossary (character names, place names, aliases, faction names — up to 100 terms) and passes it to Deepgram with every streaming connection. When the glossary is edited mid-session, the backend uses Deepgram's Configure control message to update keyterms on live streams without reconnecting.

### 3.3 Echo prevention
When the bot is playing TTS, the STT pipeline ignores audio from the bot's own user ID. A `bot_is_speaking` flag managed by the bot gates the Deepgram streaming for the bot's user ID as a belt-and-suspenders measure against mic bleed.

## 4. Data Model (Draft)

### users
```
id, email, platform_role (admin|user), subscription_tier (free|pro),
created_at, elevenlabs_chars_used_this_period
```

### campaigns
```
id, name, description, created_by (user_id), invite_code (unique),
created_at, updated_at
```

### campaign_members
```
id, campaign_id, user_id, campaign_role (dm|player), joined_at
```

### sessions
```
id, campaign_id, title, status (active|ended),
started_at, ended_at, dm_session_notes
```

### transcript_entries (raw log — ephemeral working material)
```
id, session_id, speaker_user_id (Discord), speaker_display_name,
text, segment_start_ts, segment_end_ts, created_at
```
> Not vectorized. Not fed to the chatbot. The raw log exists only to give the DM source material to curate from.

### memory_entries (campaign memory — the canonical record)
Campaign-level, optionally tagged to a session. Single table with a `kind` discriminator.
```
id, campaign_id, session_id (nullable),
kind (note | response | event),
visibility (public | dm_only),
source_timestamp,         -- earliest source transcript line's ts, for ordering
created_at,               -- when DM created/promoted

-- source context (optional)
selected_transcript_ids (array, nullable),
dm_annotation (nullable),
linked_glossary_ids (array),

-- response-only fields
character_id (nullable),
additional_context (nullable),
generated_text (nullable),
final_text (nullable),
audio_file_ref (nullable),
queue_position (nullable),
queue_status (nullable),  -- pending|playing|played|cancelled
played_at (nullable),

-- vector tracking
qdrant_point_id (nullable),
vector_updated_at (nullable)
```

**Entry kinds:**
- `note` — DM promoted transcript lines, optionally with annotation
- `response` — AI-generated NPC dialogue with full audit trail
- `event` — DM-authored standalone entry (e.g., "combat with bandits, party won, took prisoner")

**Ordering rule:** memory log UI orders by `source_timestamp` when present, falling back to `created_at`. This makes campaign memory read as a condensed story-order replay rather than DM-action order.

### characters
```
id, campaign_id, name, description, personality, speech_notes,
elevenlabs_voice_id, secrets, linked_glossary_ids,
qdrant_point_id, vector_updated_at, created_at
```

### glossary_entries
```
id, campaign_id, type (character|place|faction|event|item|lore|rule|other),
name, aliases (array), description,
known_by_character_ids (array),
linked_entry_ids (array), tags (array),
qdrant_point_id, vector_updated_at,
created_at, updated_at
```

### campaign_files
DM-uploaded reference material attached to a campaign — handouts, maps, PDFs of homebrew rules, inspiration images, session prep notes, etc. Phase 1 is storage-only; AI ingestion is deferred.
```
id, campaign_id, uploaded_by (user_id),
filename, display_name (editable),
mime_type, file_size_bytes,
storage_path,              -- Supabase Storage reference
file_kind (docx|txt|md|pdf|image|other),
description (nullable),    -- DM's freeform note about the file
tags (array),
-- future AI ingestion fields (nullable in v1)
ingestion_status (pending|ingested|skipped|failed, default skipped),
qdrant_point_ids (array, nullable),
extracted_text (nullable),
created_at, updated_at
```

**Supported file types for v1:** `.docx`, `.txt`, `.md`, `.pdf`, and common image formats (`.png`, `.jpg`, `.jpeg`, `.gif`, `.webp`).

**Storage:** files live in Supabase Storage under a per-campaign path (`campaigns/{campaign_id}/files/{file_id}/{filename}`). Access is gated by campaign membership via RLS.

**Phase 1 behavior:** upload, list, rename, describe, tag, download, delete. That's it. No preview pane required for v1 but worth having for images and PDFs if cheap.

**Future AI ingestion (Phase 3+):** the schema anticipates extracting text from text-based files (docx/txt/md/pdf), vectorizing into Qdrant, and letting the chatbot and response generator reference uploaded material. Images would require a vision model pass to be searchable — out of scope for now but the `file_kind` distinction keeps that door open. The `ingestion_status` field lets the DM opt files in/out of AI reference on a per-file basis when that feature ships.

### Qdrant collections
Single collection per campaign (or a shared collection with `campaign_id` in the payload filter — TBD during implementation based on Qdrant performance at scale). Vectorized content:
- All `memory_entries` (note, response, event)
- All `characters`
- All `glossary_entries`

**Vectorization rules:**
- **On create**: append-vectorize immediately so new entries are searchable right away.
- **On edit**: re-vectorize only the edited entry (upsert by `qdrant_point_id`). Linked entries are not cascaded — they still reference the updated entry at query time.
- **On soft-delete**: remove from Qdrant; keep row in Postgres with a `deleted_at` marker.

### Derivations / views
- **Audio queue view:** `memory_entries where kind=response and queue_status in (pending, playing) order by queue_position`
- **Whisper initial_prompt:** concatenated names + aliases from all glossary entries in the active campaign

## 5. Prompt Assembly for NPC Response Generation

This is the three-layer model. When the DM generates a response, the backend automatically assembles:

**Layer 1 — Immediate context**
- DM-selected transcript lines from the current session
- DM's optional "additional context" freeform note from the popup

**Layer 2 — Character knowledge**
- Character sheet: name, description, personality, speech notes, secrets
- Depth-1 resolution of the character's `linked_glossary_ids` (places they know, factions they belong to, items they carry)

**Layer 3 — Historical context (automatic RAG)**
- Backend queries Qdrant with the character's name + key identifiers
- Retrieves top 3 most semantically relevant memory entries that involve this character across the campaign's history
- Includes a "time since last encounter" note derived from session timestamps ("It has been 4 sessions / ~2 months of real time since you last met this party")

**Full prompt structure:**
```
System: You are voicing {Character} in a tabletop RPG session. Stay strictly
in character. Never reveal information the character would not know. Keep
responses to 1–3 sentences unless the scene requires more. Never advance the
plot without DM direction. Output only dialogue — no stage directions.

[Character sheet]
{name, description, personality, speech_notes, secrets}

[Linked knowledge the character possesses]
{depth-1 glossary entries linked from the character}

[Previous encounters with this party]
{top 3 Qdrant hits from campaign memory}
{time-since-last-encounter note}

[Immediate scene]
{DM-selected transcript lines}

[DM guidance]
{additional_context if provided}

Respond as {Character} would to the above.
```

**Information containment is structural, not prompt-based.** If a glossary entry isn't linked to the character via `known_by_character_ids` or `linked_entry_ids`, it never enters the prompt. The DM controls what each NPC knows by explicit linkage. This is how "AI revealing too much or making things up" is prevented — not by asking Claude nicely, but by not giving Claude the information in the first place.

**Top-N is tunable.** Start with top 3 historical hits. DM can adjust later if results are too sparse or too noisy.

## 6. Campaign-Memory Chatbot

### 6.1 What it is
A natural-language query interface over the campaign's curated memory. The DM asks questions, the chatbot answers from what's actually been recorded in the campaign.

### 6.2 What it searches
- All `memory_entries` (note, response, event) for the campaign
- All `glossary_entries` for the campaign
- All `characters` for the campaign

### 6.3 What it does NOT search
- Raw transcript entries (too noisy, irrelevant table talk, mistranscriptions)
- Other campaigns
- DM-only entries when accessed by a player (future — DM-only in v1)

### 6.4 Query flow
1. DM types a question in the chatbot panel.
2. Backend embeds the query and runs a Qdrant similarity search against the campaign's collection, filtered by `campaign_id`.
3. Top-K hits (start with K=8) are retrieved with full payloads.
4. Claude is called with:
   ```
   System: You are a helpful campaign historian. Answer the DM's question
   based only on the retrieved campaign memory below. If the answer isn't
   in the retrieved context, say so. Cite which entries you drew from.

   [Retrieved entries with IDs and types]

   DM question: {query}
   ```
5. Response streams back to the dashboard.
6. Each cited entry is clickable — jumps to that entry in the memory log.

### 6.5 Why not ingest the raw transcript
The DM's explicit design choice: the raw log is filled with dice rolls, rules arguments, bathroom-break chatter, and mistranscribed proper nouns. Vectorizing it would drown the signal. The curation step is what turns noise into memory. The chatbot is only as good as the curation, which is exactly the incentive we want.

### ElevenLabs is still the biggest line item
This is the cost that scales most aggressively with usage. Every user needs a hard character cap, tracked in `users.elevenlabs_chars_used_this_period` and checked **before every TTS call**. The dashboard should show remaining budget.

### Deepgram is the second budget to watch
A 4-hour session with 2–3 hours of active speech across all speakers costs roughly $1 per session. For a weekly campaign that's ~$4–$5/month per DM. At Phase 2 scale with 20 active DMs, Deepgram costs run ~$100/month total. Free tier users need a cap on STT minutes per period, just like ElevenLabs chars.

### Future cloud infrastructure (deferred)
- Supabase Pro ($25/mo) when storage/bandwidth demands it
- Possibly managed Qdrant Cloud if self-hosting becomes a burden
- **No GPU hosting required** — Deepgram handles STT compute as a service

## 8. Known Risks & Limitations

### High-impact
1. **Open-mic table audio is noisy.** Mitigated by the curation model — raw log is disposable, only what the DM promotes matters.
2. **Fantasy proper nouns.** Mitigated via Deepgram Keyterm Prompting, which is a contextual in-context adapter (not just keyword boosting). Up to 100 keyterms per stream, updatable mid-session. Significantly better than Whisper's `initial_prompt` approach.
3. **ElevenLabs costs can spiral.** Hard caps enforced before every TTS call.
4. **Deepgram costs scale with usage.** Hard caps on STT minutes per user per period. Must be enforced at session start and tracked during session. Silero VAD on our side is non-negotiable — without it, silence gets billed.
5. **Bot hearing itself.** Suppress bot's own audio in STT during playback.
6. **Third-party audio dependency.** Audio is sent to Deepgram for transcription. Must be disclosed in the Discord recording consent notice. Privacy-conscious users may object.

### Medium-impact
7. **Response generation latency floor.** Claude (2–6s) + ElevenLabs Flash v2.5 streaming (~75–400ms TTFB) + DM review time. Must use Flash v2.5 + streaming from day one.
8. **Deepgram network dependency.** A Deepgram outage means sessions can't transcribe. No offline fallback in v1. Acceptable for a non-critical tool; document the limitation in status page.
9. **Raw audio deleted on session end** means no re-transcription with better models later. Accepted for v1.
10. **Chatbot quality depends on curation quality.** If the DM doesn't curate diligently, the chatbot has nothing to work with. This is by design but worth flagging — users will need a gentle onboarding nudge about the curation habit.
11. **Qdrant re-vectorization on edit is per-entry only.** Editing a character doesn't automatically update response entries that referenced that character. Usually fine; occasionally a staleness issue.

### Low-impact / deferred
12. **Multi-tenancy isolation.** Supabase RLS handles this. Schema is already campaign-keyed throughout.
13. **Discord TOS around recording.** Bot posts a notice message on join (including Deepgram disclosure); per-campaign player consent flow in Phase 2+.
14. **Vendor lock-in on Deepgram.** The STT layer is well-isolated. If Deepgram changes pricing or terms, AssemblyAI and Gladia are viable swap-in alternatives.

## 10. Open Technical Questions

1. **Node ↔ Python websocket protocol** — message schema for audio segments, session lifecycle, playback commands.
2. **Silero VAD tuning** — thresholds for gating Deepgram streams on D&D audio. Too aggressive = missed speech. Too loose = paying for silence.
3. **Supabase Realtime vs. dedicated websocket** for transcript streaming — benchmark with 2–5 concurrent speakers.
4. **Deepgram connection strategy** — one long-lived connection per user for the session vs. open/close per utterance. Long-lived is simpler and Deepgram bills on audio sent, not connection duration, so likely the right default.
5. **Keyterm list management** — glossary may exceed Deepgram's 100-keyterm limit per stream. Prioritize character names and recently-edited entries. Worth implementing a priority scoring function.
6. **Claude prompt caching** for character/glossary layers — meaningfully reduces per-generation cost and latency.
7. **ElevenLabs streaming → Discord audio pipeline** — Discord voice needs Opus frames, ElevenLabs streams MP3/PCM. Transcode via `prism-media` in the Node bot.
8. **Qdrant collection strategy** — one collection per campaign vs. one shared collection with `campaign_id` payload filter. Depends on expected scale and Qdrant's filter performance.
9. **Embedding model choice** — Jake uses Nomic Embed in ExpertAI, reasonable default here. Worth revisiting if retrieval quality on narrative text specifically is weak.
10. **Chatbot context budget** — top-K retrieval count and per-entry token budget. Start K=8, tune from real use.
11. **Row-level security policies** on Supabase for multi-campaign isolation.
12. **Session pause behavior** — paused time counted toward session duration? Close Deepgram connections on pause or just stop sending audio? (Probably: close connections to save reconnect latency is minimal.)

## 12. Decisions Locked In

- **Product framing:** storytelling and memory tool, not stat tracking
- Node.js bot + Python backend + React + Supabase + Qdrant
- **Deepgram Nova-3 streaming for STT** with Keyterm Prompting driven by the campaign glossary
- **Silero VAD on CPU** as a cost-control filter before Deepgram
- **No GPU hardware required** at any phase; everything runs on cheap CPU hosting
- Per-session raw logs, campaign-level memory/glossary/characters
- Raw audio deleted immediately on session end
- **Raw transcript is ephemeral working material, not ingested into Qdrant**
- **Campaign memory is the canonical record and the only thing the chatbot sees**
- Memory log ordered by source transcript timestamp when available
- Three entry kinds in campaign memory: `note`, `response`, `event`
- Standalone `event` entries allow DMs to log things like combat without raw transcript
- DM-only visibility field on memory entries for hidden canon
- Pause STT for noise control (no OOC tagging needed at capture)
- Two-step response flow: generate text → DM edits → generate audio → queue → DM plays
- Pre-made ElevenLabs voices only (no cloning in v1)
- **Three-layer prompt assembly for response generation**: immediate context + character knowledge + historical context from Qdrant
- **Historical context retrieval is automatic** (not manual DM selection), top 3 by relevance
- **Campaign-memory chatbot is in Phase 1**, DM-only
- Append-vectorize on create, re-vectorize on edit (entry only, no cascade)
- Platform roles: admin/user. Campaign roles: dm/player. Subscription tiers: free/pro.
- Players see both logs during early testing (schema supports locking this down later)
- Information containment via explicit glossary linkage, not prompt instructions
- **Campaign file uploads in Phase 1 as storage-only** (docx, txt, md, pdf, images), with schema designed for future AI ingestion
- **Two environments from Phase 1.5 onward**: `dev.fablescribe.io` and `fablescribe.io` with full data/secret isolation
- **Hosting**: Vercel (frontend), Railway (backend + bot on cheap CPU instances), Supabase (auth/db/realtime/storage), Qdrant Cloud for dev + self-hosted for prod, **Deepgram cloud STT with separate dev/prod API keys**
- **Deploy flow**: push to `dev` branch auto-deploys to dev env; push to `main` auto-deploys to prod
